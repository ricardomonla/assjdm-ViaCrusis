## Modelo de datos

### Tabla SQLite: `personas`

| Campo | Tipo | Descripción |
|-------|------|-------------|
| `id` | INTEGER PK | Auto-increment |
| `nombre` | TEXT | Nombre |
| `apellido` | TEXT | Apellido |
| `dni` | TEXT | Documento (opcional) |
| `telefono` | TEXT | Contacto (opcional) |
| `created_at` | TEXT | Timestamp de creación |

UNIQUE: `(nombre, apellido)` — evita duplicados.

### Tabla SQLite: `roles` (catálogo)

| Campo | Tipo | Descripción |
|-------|------|-------------|
| `id` | TEXT PK | Código único: `actor`, `staff`, `sonido`, etc. |
| `nombre` | TEXT | Nombre para mostrar |
| `icono` | TEXT | Emoji para la lista |

Siembra inicial:

| id | nombre | icono |
|----|--------|-------|
| `actor` | Actor/Actriz | 🎭 |
| `staff` | Staff | 🔧 |
| `sonido` | Sonido/Técnica | 🎵 |
| `donante` | Donante/Colaborador | 🤝 |
| `logistica` | Logística | 🚚 |
| `otro` | Otro | ⭐ |

### Tabla SQLite: `persona_roles` (N:M)

| Campo | Tipo | Descripción |
|-------|------|-------------|
| `persona_id` | INTEGER FK | Referencia a `personas.id` |
| `rol_id` | TEXT FK | Referencia a `roles.id` |
| `personaje` | TEXT | Nombre del personaje asignado (solo para rol actor) |

PK compuesta: `(persona_id, rol_id)`. Una persona elige los roles que quiera de la lista.

### Feature 3: Acciones Director

Solo visible para Director:
- Editar persona (nombre, teléfono, roles).
- Eliminar persona con confirmación.

### Feature 4: Placeholders desde personajes

Siembra automática desde tabla `cues`: cada personaje único (JESÚS, PILATOS, etc.)
se crea como persona temporal con nombre = personaje y rol = `actor` + `personaje` = nombre.

### Feature 5: Endpoint de verificación (status)

**Endpoint:** `GET /personas/api.php?action=status`

**Respuesta:**
```json
{
  "ok": true,
  "status": "healthy",
  "version": "26.12",
  "counts": {
    "personas": 27,
    "roles": 6,
    "persona_roles": 27
  },
  "migration": {
    "done": true,
    "has_real_data": true
  },
  "last_person_created": "2026-04-18 00:22:02",
  "timestamp": "2026-04-18 01:00:00"
}
```

**Usos:**
- Verificar que la migración se completó
- Monitorear cantidad de personas registradas
- Debug de problemas en producción

### Feature 6: Mecanismo de migración DB (Test → Prod)

**Archivos:**
- `tools/db_export.php` — Exporta DB local a JSON
- `data/db_import.php` — Importa JSON en producción (vía webhook)
- `data/migration_personas.json` — Datos a migrar (se commitea)
- `docs/DEPLOY_MIGRATION.md` — Documentación completa

**Flujo:**
1. `docker exec vcby-test php /app/tools/db_export.php`
2. `git add data/migration_personas.json && git commit && git push`
3. Webhook en producción ejecuta `db_import.php` automáticamente

**Flujo:**
1. `ensureSchema()` detecta `personas` vacía → extrae personajes de `cues`.
2. Crea persona "JESÚS" con rol `actor` y `personaje = 'JESÚS'`.
3. En la lista aparece: **JESÚS** — 🎭 Actor-Jesús ⚠️ (falta datos reales).
4. Alguien clickea 📝 → completa nombre real, apellido, DNI, teléfono.
5. Se guarda → ahora muestra: **Juan Pérez** — 🎭 Actor-Jesús ✅.
6. El nombre del personaje queda en `persona_roles.personaje`, no se pierde.

```
Antes:   JESÚS (placeholder) → 🎭 Actor-Jesús  ⚠️ Falta: Apellido, DNI, Tel
Después: Juan Pérez          → 🎭 Actor-Jesús  ✅
```
